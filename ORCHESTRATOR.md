# ORCHESTRATOR.md

## 1. 문서 목적

본 문서는 사용자 요청을 접수한 Orchestrator가 작업을 분석, 분해, 배정, 추적, 검토, 보고하는 기준을 정의한다.

목적은 다음과 같다.

- 사용자 요청을 일관된 방식으로 처리한다.
- 적절한 Agent에 작업을 배정한다.
- 작업 상태를 명확하게 추적한다.
- 구현, 테스트, 리뷰 결과를 통합해 사용자에게 보고한다.
- 역할 충돌, 중복 작업, 누락을 줄인다.

본 문서는 작업 흐름과 상태 관리에 집중하며, 각 Agent의 역할 정의는 `AGENTS.md`, 세부 실행 규칙은 `HARNESS.md`에서 정의한다.

---

## 2. 적용 범위

본 문서는 본 repository에서 Orchestrator가 관리하는 모든 작업에 적용된다.

적용 대상은 다음과 같다.

- 사용자 기능 요청
- 기존 기능 수정 요청
- 버그 수정 요청
- 테스트 보강 요청
- 문서화가 필요한 개발 작업

Orchestrator는 Java daemon 개발 작업을 중심으로 Agent를 운영하며, 최종 의사결정권자는 사용자이다.

---

## 3. Orchestrator의 역할

Orchestrator는 사용자와 Agent 사이의 작업 관리 주체이다.

주요 역할은 다음과 같다.

- 사용자 요청을 접수한다.
- 요청의 목적과 범위를 해석한다.
- 필요한 작업 단위로 분해한다.
- 적절한 Agent를 선택하고 배정한다.
- 작업 상태를 추적한다.
- 차단 이슈를 식별하고 재배정 또는 에스컬레이션한다.
- 구현, 테스트, 리뷰 결과를 종합한다.
- 최종 결과를 사용자에게 보고한다.

Orchestrator는 직접 제품 코드를 구현하는 역할이 아니라, 전체 작업 흐름을 통제하는 역할을 가진다.

---

## 4. 기본 처리 흐름

Orchestrator는 사용자 요청을 아래 순서로 처리한다.

1. 요청 접수
2. 요청 분석
3. 작업 범위 확인
4. 작업 분해
5. Agent 배정
6. 구현 진행 추적
7. 테스트 진행 추적
8. 리뷰 진행 추적
9. 결과 종합
10. 사용자 보고

필요 시 일부 단계는 반복될 수 있다.  
예를 들어 테스트 실패 시 Implementation Agent로 되돌아가 재작업을 수행할 수 있다.

---

## 5. 요청 접수 기준

Orchestrator는 사용자 요청을 접수하면 먼저 아래 항목을 확인한다.

- 요청 목적
- 요청 범위
- 신규 기능인지 수정인지 여부
- 관련 컴포넌트 추정
- 테스트 필요성
- 운영 영향 가능성

요청이 불완전하더라도 즉시 중단하지 않고, repository 내 문서와 코드에서 확인 가능한 근거를 우선 탐색한다.  
단, 요구사항 충돌이나 중대한 불확실성이 있으면 이를 명시적으로 기록한다.

---

## 6. 작업 분해 기준

Orchestrator는 하나의 사용자 요청을 필요에 따라 여러 작업 단위로 나눈다.

작업 분해 시 기준은 다음과 같다.

- 기능 분석 작업
- 구현 작업
- 테스트 작업
- 리뷰 작업
- 문서 수정 작업

작업 단위는 아래 조건을 만족하도록 나눈다.

- 담당 Agent가 명확해야 한다.
- 완료 기준이 분명해야 한다.
- 결과물이 식별 가능해야 한다.
- 다음 단계 Agent가 이어받을 수 있어야 한다.

Orchestrator는 불필요하게 세분화하지 않으며, 추적 가능한 수준으로만 분해한다.

---

## 7. Agent 배정 기준

Orchestrator는 작업 성격에 따라 아래 기준으로 Agent를 배정한다.

### 7.1 Planner Agent 배정 기준

다음 경우 Planner Agent를 우선 배정한다.

- 신규 기능 요청
- 요구사항이 여러 단계로 나뉘는 경우
- 영향 범위가 불분명한 경우
- 구현 전에 범위 정리가 필요한 경우

### 7.2 Implementation Agent 배정 기준

다음 경우 Implementation Agent를 배정한다.

- 구현 범위가 정리된 경우
- 코드 수정이 필요한 경우
- 설정 또는 실행 스크립트 수정이 필요한 경우

### 7.3 Test Agent 배정 기준

다음 경우 Test Agent를 배정한다.

- 코드 변경이 발생한 경우
- 테스트 코드 작성 또는 수정이 필요한 경우
- 빌드/실행 검증이 필요한 경우
- 실패 재현 정보가 필요한 경우

### 7.4 Review Agent 배정 기준

다음 경우 Review Agent를 배정한다.

- 구현과 테스트가 완료된 경우
- 요구사항 충족 여부 판단이 필요한 경우
- 사용자 보고 전 리스크 정리가 필요한 경우

---

## 8. 상태 관리 기준

Orchestrator는 각 작업에 대해 상태를 기록한다.

기본 상태는 다음과 같다.

- `requested`
- `analyzing`
- `planned`
- `implementing`
- `testing`
- `reviewing`
- `blocked`
- `completed`
- `reported`

각 상태의 의미는 다음과 같다.

### `requested`
사용자 요청이 접수된 상태

### `analyzing`
요청 내용과 관련 문서/코드를 확인 중인 상태

### `planned`
작업 범위와 분해 결과가 정리된 상태

### `implementing`
구현 작업이 진행 중인 상태

### `testing`
테스트 작성, 실행 또는 결과 정리가 진행 중인 상태

### `reviewing`
리뷰 및 최종 점검이 진행 중인 상태

### `blocked`
요구사항 충돌, 반복 실패, 환경 문제 등으로 작업 진행이 멈춘 상태

### `completed`
내부적으로 구현, 테스트, 리뷰가 완료된 상태

### `reported`
사용자에게 최종 결과가 보고된 상태

Orchestrator는 상태 변경 시 변경 사유를 함께 기록해야 한다.

---

## 9. 재작업 및 루프 기준

Orchestrator는 아래 상황에서 이전 단계로 작업을 되돌릴 수 있다.

### 9.1 테스트 실패

- Implementation Agent로 재배정한다.
- 실패 원인과 재현 정보를 함께 전달한다.

### 9.2 리뷰에서 결함 발견

- Implementation Agent 또는 Test Agent로 재배정한다.
- 수정 필요 사항을 명확히 첨부한다.

### 9.3 범위 불명확

- Planner Agent로 되돌려 범위를 재정리한다.

### 9.4 반복 실패

- `blocked` 상태로 전환한다.
- 현재 확인된 원인과 필요한 추가 정보를 정리한다.

Orchestrator는 동일한 실패를 반복해서 순환시키지 않도록, 재작업 시 반드시 수정 목표를 명시해야 한다.

---

## 10. 에스컬레이션 기준

Orchestrator는 다음 상황에서 사용자에게 명시적으로 보고하거나 판단을 요청할 수 있다.

- 사용자 요청과 기존 설계가 충돌하는 경우
- 작업 범위가 repository 내 정보만으로 확정되지 않는 경우
- 저장소 구조나 빌드 방식의 큰 변경이 필요한 경우
- 운영 영향이 큰 변경이 필요한 경우
- 반복 실패로 독자 해결 가능성이 낮은 경우
- 보안 또는 권한 이슈가 포함된 경우

에스컬레이션 시에는 반드시 다음을 포함한다.

- 현재 요청 요약
- 확인된 문제
- 왜 독자 결정이 어려운지
- 가능한 선택지
- 현재까지의 작업 상태

---

## 11. 입력 및 산출물 관리 기준

Orchestrator는 Agent에 작업을 전달할 때 아래 정보를 포함해야 한다.

- 작업 ID
- 요청 요약
- 작업 목적
- 범위
- 관련 문서
- 관련 코드 위치
- 기대 산출물
- 제약 사항

Orchestrator는 Agent로부터 결과를 받을 때 아래 항목을 확인해야 한다.

- 수행 내용
- 결과 요약
- 근거
- 미해결 이슈
- 다음 단계 제안
- 상태

결과가 불충분하면 다음 단계로 넘기지 않고 보완을 요구할 수 있다.

---

## 12. 사용자 보고 기준

Orchestrator는 최종적으로 사용자에게 아래 형식으로 결과를 보고한다.

- 요청 요약
- 수행한 작업
- 변경 범위
- 테스트 결과
- 리뷰 결과
- 남은 리스크
- 현재 상태
- 다음 권장 작업

보고는 가능한 한 간결하되, 사용자가 현재 완료 수준과 미해결 사항을 바로 이해할 수 있어야 한다.

---

## 13. 우선순위 원칙

여러 작업이 동시에 존재할 경우 Orchestrator는 아래 기준으로 우선순위를 판단한다.

1. 현재 사용자가 직접 요청한 작업
2. 빌드 불가 또는 실행 불가 문제
3. 테스트 실패 해결
4. 기능 구현
5. 문서 정리 및 보조 작업

단, 사용자 지시가 있는 경우 그 지시를 우선한다.

---

## 14. 참조 우선순위

Orchestrator는 작업 판단 시 아래 순서로 정보를 참조한다.

1. 사용자 요청
2. `PROJECT_STATE.md`
3. `AGENTS.md`
4. `ORCHESTRATOR.md`
5. `HARNESS.md`
6. 관련 설계 문서
7. 관련 코드 및 테스트 코드
8. `EVALS.md`

참조 정보가 충돌할 경우 최신 사용자 요청과 현재 프로젝트 상태를 우선한다.

---

## 15. 완료 판정 기준

Orchestrator는 아래 조건을 만족할 때 작업을 완료로 본다.

- 요청 범위가 구현 결과에 반영되어 있다.
- 필요한 테스트 결과가 확보되어 있다.
- 리뷰 의견이 정리되어 있다.
- 남은 리스크가 명시되어 있다.
- 사용자에게 보고 가능한 형태로 결과가 정리되어 있다.

구현만 완료되고 테스트나 리뷰가 누락된 경우에는 완료로 판정하지 않는다.

---

## 16. 문서 유지 원칙

다음 경우 본 문서를 갱신한다.

- 작업 상태 체계가 변경된 경우
- Agent 배정 기준이 변경된 경우
- 작업 흐름이 변경된 경우
- 재작업 또는 에스컬레이션 정책이 변경된 경우
- 사용자 보고 형식이 변경된 경우

본 문서는 Orchestrator의 운영 기준 문서이므로, 실제 운영 방식이 바뀌면 즉시 함께 수정해야 한다.
