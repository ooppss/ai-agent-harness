# task-flow.md

## 1. 문서 목적

본 문서는 사용자 요청이 접수된 이후 Orchestrator와 Agent가 작업을 수행하는 전체 흐름을 정의한다.

본 문서의 목적은 다음과 같다.

- 사용자 요청이 어떤 단계로 처리되는지 명확히 한다.
- Orchestrator와 Agent의 작업 흐름을 일관되게 유지한다.
- 구현, 테스트, 리뷰, 보고의 연결 관계를 정리한다.
- 재작업과 차단 상태 처리 흐름을 정의한다.
- 작업 추적과 후속 개선의 기준을 제공한다.

본 문서는 역할 정의가 아니라 작업 순서를 설명하는 문서이다.  
Agent 역할은 `AGENTS.md`, 실행 규칙은 `HARNESS.md`, 상태 관리는 `ORCHESTRATOR.md`를 따른다.

---

## 2. 작업 흐름 개요

본 프로젝트의 기본 작업 흐름은 다음과 같다.

1. 사용자 요청 접수
2. 요청 분석
3. 작업 범위 확인
4. 작업 분해
5. Agent 배정
6. 구현 수행
7. 테스트 수행
8. 리뷰 수행
9. 결과 종합
10. 사용자 보고

필요 시 테스트 실패, 리뷰 지적, 범위 불명확 등의 사유로 이전 단계로 되돌아간다.

---

## 3. 시작 조건

작업은 다음 조건이 충족되면 시작할 수 있다.

- 사용자 요청이 존재한다.
- 현재 프로젝트 상태 문서(`PROJECT_STATE.md`)가 확인 가능하다.
- 운영 문서(`AGENTS.md`, `ORCHESTRATOR.md`, `HARNESS.md`)가 존재한다.
- 작업 대상 repository에 접근 가능하다.

위 조건이 충족되지 않으면 Orchestrator는 현재 확인 가능한 범위를 먼저 정리하고, 부족한 정보를 명시한다.

---

## 4. 단계별 작업 흐름

### 4.1 사용자 요청 접수

작업은 사용자가 Orchestrator에게 기능 구현, 수정, 테스트 보강, 문서 정리 등의 요청을 전달하면서 시작된다.

입력:
- 사용자 요청

출력:
- 접수된 작업 항목
- 초기 요청 요약

Orchestrator는 요청 접수 시 다음을 우선 확인한다.

- 요청 목적
- 요청 범위
- 신규 기능인지 수정인지 여부
- 관련 컴포넌트 추정
- 테스트 필요성
- 운영 영향 가능성

---

### 4.2 요청 분석

Orchestrator는 접수된 요청을 분석한다.

주요 작업:
- 사용자 요청 해석
- 관련 문서 및 코드 범위 추정
- 현재 프로젝트 상태와의 정합성 확인
- 범위 불명확 여부 확인

참조 대상:
- 사용자 요청
- `PROJECT_STATE.md`
- `AGENTS.md`
- `ORCHESTRATOR.md`
- `HARNESS.md`
- 관련 설계 문서
- 관련 코드 및 테스트 코드

출력:
- 분석 결과
- 초기 범위 정의
- 불확실성 또는 충돌 사항

---

### 4.3 작업 범위 확인

Orchestrator는 분석 결과를 바탕으로 실제 작업 범위를 정리한다.

주요 작업:
- 변경 대상 식별
- 관련 문서/코드 위치 확인
- 영향 범위 정리
- 작업 단위 분해 가능 여부 판단

출력:
- 작업 범위 정의
- 관련 파일/폴더 목록
- 예상 후속 단계

범위가 불명확하면 Planner Agent를 우선 배정할 수 있다.

---

### 4.4 작업 분해

Orchestrator는 필요 시 요청을 여러 작업 단위로 나눈다.

작업 분해 기준:
- 분석 작업
- 구현 작업
- 테스트 작업
- 리뷰 작업
- 문서 수정 작업

분해 원칙:
- 담당 Agent가 명확해야 한다.
- 완료 기준이 분명해야 한다.
- 다음 단계로 결과 인계가 가능해야 한다.

출력:
- 작업 단위 목록
- 작업별 목적
- 작업별 기대 산출물

---

### 4.5 Agent 배정

Orchestrator는 작업 성격에 따라 Agent를 배정한다.

기본 배정 기준:
- 범위 정리 필요 시 → Planner Agent
- 코드 수정 필요 시 → Implementation Agent
- 테스트 필요 시 → Test Agent
- 검토 및 최종 점검 필요 시 → Review Agent

출력:
- 작업별 담당 Agent
- 작업 입력 정보
- 기대 산출물
- 현재 상태

Agent는 배정 후 작업 시작 전에 관련 문서와 코드를 검토해야 한다.

---

### 4.6 구현 수행

Implementation Agent는 배정된 구현 작업을 수행한다.

주요 작업:
- 관련 코드 검토
- Java daemon 코드 작성 또는 수정
- 필요한 설정 파일 또는 스크립트 수정
- 변경 사항 요약

입력:
- 작업 범위
- Planner Agent 산출물
- 관련 코드 및 설정
- `PROJECT_STATE.md`

출력:
- 변경된 코드
- 변경 요약
- 테스트 필요 항목
- 미해결 이슈

구현 결과는 다음 단계인 테스트 작업으로 전달된다.

---

### 4.7 테스트 수행

Test Agent는 구현 결과를 검증한다.

주요 작업:
- 테스트 포인트 확인
- 테스트 코드 작성 또는 수정
- 빌드 및 테스트 실행
- 실패 재현 정보 정리
- 테스트 결과 기록

입력:
- Implementation Agent 결과
- 테스트 코드
- 실행 스크립트
- 수용 기준 초안

출력:
- 테스트 결과
- 통과 항목
- 실패 항목
- 재현 정보
- 테스트 관점 리스크

테스트 통과 시 리뷰 단계로 이동한다.  
테스트 실패 시 재작업 흐름으로 이동한다.

---

### 4.8 리뷰 수행

Review Agent는 구현 및 테스트 결과를 검토한다.

주요 작업:
- 요구사항 충족 여부 확인
- 테스트 결과 해석
- 품질상 우려 식별
- 남은 리스크 정리
- 사용자 보고용 검토 의견 작성

입력:
- Planner Agent 산출물
- Implementation Agent 결과
- Test Agent 결과
- 관련 코드 및 문서

출력:
- 리뷰 의견
- 요구사항 충족 여부 판단
- 품질상 우려 사항
- 남은 리스크
- 최종 보고용 요약

리뷰 결과가 적절하면 결과 종합 단계로 이동한다.  
리뷰에서 수정 필요 사항이 발견되면 재작업 흐름으로 이동한다.

---

### 4.9 결과 종합

Orchestrator는 구현, 테스트, 리뷰 결과를 종합한다.

주요 작업:
- 각 단계 결과 정리
- 완료 수준 확인
- 남은 리스크 확인
- 최종 상태 판정
- 사용자 보고 내용 구성

출력:
- 최종 결과 요약
- 상태 판정
- 보고용 정리 자료

---

### 4.10 사용자 보고

Orchestrator는 작업 결과를 사용자에게 보고한다.

보고 항목은 다음을 포함한다.

- 요청 요약
- 수행한 작업
- 변경 범위
- 테스트 결과
- 리뷰 결과
- 남은 리스크
- 현재 상태
- 다음 권장 작업

보고 완료 시 작업은 `reported` 상태로 간주한다.

---

## 5. 재작업 흐름

다음 상황에서는 이전 단계로 되돌아간다.

### 5.1 테스트 실패

흐름:
- Test Agent 결과 확인
- 실패 원인 정리
- Implementation Agent로 재배정
- 수정 후 다시 테스트 수행

### 5.2 리뷰 지적 사항 발생

흐름:
- Review Agent 의견 확인
- 수정 필요 항목 정리
- Implementation Agent 또는 Test Agent로 재배정
- 수정 후 다시 리뷰 수행

### 5.3 범위 불명확

흐름:
- 현재 범위 정리
- Planner Agent로 재배정
- 범위 재정의 후 구현 단계로 이동

재작업은 단순 반복이 아니라, 이전 실패 원인을 반영한 수정 작업이어야 한다.

---

## 6. 차단 흐름

다음 경우 작업은 차단 상태로 전환될 수 있다.

- 요구사항 충돌
- 반복 실패
- 환경 문제
- 저장소 구조상 제약
- 운영 영향이 큰 변경 필요
- 보안 또는 권한 문제

차단 시 Orchestrator는 다음을 기록한다.

- 현재 작업 요약
- 차단 원인
- 현재까지의 진행 상태
- 독자 해결 가능 여부
- 필요한 후속 조치 또는 사용자 판단 항목

차단 상태의 작업은 문제 해소 전까지 다음 단계로 진행하지 않는다.

---

## 7. 상태 전이 개요

기본 상태 전이는 다음과 같다.

```text
requested
  → analyzing
  → planned
  → implementing
  → testing
  → reviewing
  → completed
  → reported

예외 흐름은 다음과 같다.

  testing → implementing
  reviewing  →  implementing
  reviewing  →  testing
  any step  →  blocked

상태 정의의 상세 기준은 ORCHESTRATOR.md를 따른다.

## 8. 단계 간 인계 원칙
각 단계의 결과는 다음 단계가 바로 사용할 수 있도록 정리되어야 한다.

인계 시 포함해야 할 항목:

- 작업 요약
- 수행 내용
- 변경 범위
- 검증 결과
- 미해결 이슈
- 다음 단계 필요 사항

결과 인계가 불충분하면 Orchestrator는 보완 후 다시 제출하도록 요구할 수 있다.

## 9. 표준 입력 및 출력
각 단계는 가능한 한 아래 형식을 따른다.

### 9.1 표준 입력

- 작업 ID
- 요청 요약
- 작업 목적
- 작업 범위
- 관련 문서
- 관련 코드 위치
- 제약 사항
- 기대 산출물

### 9.2 표준 출력

- 작업 요약
- 수행 내용
- 결과
- 변경 대상
- 검증 또는 테스트 결과
- 미해결 이슈
- 다음 단계 권장 사항
- 현재 상태

## 10. 작업 흐름 적용 원칙
본 문서의 흐름은 기본 표준이며, 모든 작업이 반드시 동일한 깊이로 모든 단계를 거칠 필요는 없다.

예를 들어 :

- 단순 문서 수정은 구현/테스트 단계가 축소될 수 있다.
- 작은 설정 수정은 Planner 단계가 매우 짧을 수 있다.
- 복잡한 기능 구현은 Planner, Test, Review 단계가 더 강화될 수 있다.

단, 다음 원칙은 유지한다.

- 요청 분석 없이 구현을 시작하지 않는다.
- 구현 결과는 검증 근거 없이 완료로 처리하지 않는다.
- 결과는 사용자 보고 가능한 형태로 정리한다.

## 11. Linux daemon 작업 적용 시 주의 사항

본 프로젝트의 주요 제품은 Linux 서버에서 동작하는 Java daemon이므로, 작업 흐름 수행 시 다음 사항을 함께 고려한다.

- 장기 실행 안정성
- 시작 및 종료 흐름
- 예외 처리
- 로그 및 추적성
- 패키징 및 실행 구조
- 운영 환경 영향

즉, 기능 구현만 완료되었다고 해서 작업이 끝난 것으로 보지 않으며, 운영형 소프트웨어 관점의 검토를 포함한다.

## 문서 유지 원칙

다음 경우 본 문서를 갱신한다.

- 작업 단계가 변경된 경우
- 재작업 또는 차단 흐름이 변경된 경우
- 상태 전이 기준이 변경된 경우
- 단계 간 인계 방식이 변경된 경우
- 제품 특성에 따른 흐름 고려사항이 변경된 경우

본 문서는 작업 흐름 기준 문서이므로, 실제 운영 흐름이 바뀌면 즉시 함께 수정한다.
