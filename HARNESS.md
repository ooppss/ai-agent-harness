# HARNESS.md

## 1. 문서 목적

본 문서는 Orchestrator와 Agent가 본 repository 내 작업을 수행할 때 따라야 하는 실행 규칙을 정의한다.

목적은 다음과 같다.

- 작업 수행 절차를 일관되게 유지한다.
- Agent가 작업 시작 전에 필요한 문서와 코드를 검토하도록 강제한다.
- 작업 범위, 권한, 기록, 재시도 기준을 통제한다.
- 불필요한 변경, 중복 작업, 통제되지 않은 실행을 방지한다.
- 구현, 테스트, 리뷰, 보고 과정이 추적 가능하도록 만든다.

본 문서는 역할 정의보다는 실행 방식에 집중하며, Agent의 책임은 `AGENTS.md`, 작업 배정과 상태 관리는 `ORCHESTRATOR.md`에서 정의한다.

---

## 2. 적용 범위

본 문서는 본 repository에서 Orchestrator가 관리하는 모든 Agent 작업에 적용된다.

적용 대상은 다음과 같다.

- 요청 분석
- 구현 작업
- 테스트 작업
- 리뷰 작업
- 문서 수정 작업
- 결과 보고를 위한 정리 작업

본 문서는 repository 내부 문서와 코드, 빌드, 테스트, 결과 기록에 대한 실행 기준을 제공한다.

---

## 3. 기본 실행 원칙

모든 작업은 아래 원칙을 따른다.

- 작업 시작 전에 관련 문서와 코드를 검토한다.
- 확인 가능한 repository 내 근거를 우선한다.
- 작업은 요청 범위 내에서만 수행한다.
- 관련 없는 변경은 하지 않는다.
- 결과는 추적 가능한 형태로 남긴다.
- 실패는 숨기지 않고 원인과 상태를 기록한다.
- 불명확하거나 민감한 사안은 Orchestrator에 에스컬레이션한다.

---

## 4. 작업 시작 전 필수 확인 절차

모든 Agent는 작업 시작 전에 아래 순서로 정보를 확인한다.

1. 사용자 요청
2. `PROJECT_STATE.md`
3. `AGENTS.md`
4. `ORCHESTRATOR.md`
5. 본 문서(`HARNESS.md`)
6. 관련 설계 문서
7. 관련 코드 및 테스트 코드

작업에 직접 관련된 문서와 코드가 확인되지 않은 상태에서는 구현 또는 수정 작업을 시작하지 않는다.

---

## 5. 표준 작업 절차

모든 작업은 아래 절차를 기본으로 한다.

### 5.1 요청 확인

- 현재 작업의 목적과 범위를 확인한다.
- 신규 기능인지, 수정인지, 버그 대응인지 구분한다.
- 관련 컴포넌트와 영향 범위를 추정한다.

### 5.2 근거 수집

- 관련 문서, 코드, 테스트, 설정 파일을 검토한다.
- 기존 구현 방식과 현재 제약 조건을 확인한다.
- 불명확한 지점은 명시적으로 표시한다.

### 5.3 작업 수행

- 요청 범위 내에서 필요한 변경만 수행한다.
- 기존 구조와 스타일을 가능한 한 유지한다.
- 대규모 구조 변경은 별도 근거 없이 수행하지 않는다.

### 5.4 검증 준비

- 변경 사항이 어떤 방식으로 검증될 수 있는지 정리한다.
- 필요한 테스트 코드 또는 실행 절차를 확인한다.

### 5.5 결과 정리

- 수행 내용
- 변경 범위
- 확인한 근거
- 미해결 이슈
- 다음 단계 필요 사항

위 항목을 다음 단계 Agent 또는 Orchestrator가 바로 사용할 수 있는 수준으로 정리한다.

---

## 6. 허용되는 작업

Agent는 아래 범위의 작업을 수행할 수 있다.

- repository 내 문서 읽기
- repository 내 코드 읽기
- 요청 범위 내 코드 수정
- 테스트 코드 작성 및 수정
- 빌드 설정 검토 및 필요한 최소 수정
- 실행 스크립트 검토 및 필요한 최소 수정
- 결과 보고서 또는 작업 요약 작성

허용 작업은 항상 현재 요청과 직접 관련되어야 한다.

---

## 7. 제한되는 작업

Agent는 아래 작업을 임의로 수행하지 않는다.

- 사용자 요청과 무관한 코드 변경
- 관련성 없는 리팩토링
- repository 전체 구조 개편
- 외부 의존성의 임의 추가 또는 교체
- 운영 환경을 전제로 한 위험한 설정 변경
- 검증 없이 기능 완료 선언
- 테스트 실패의 임의 무시
- 요구사항의 임의 변경

위 작업이 필요하다고 판단되면 Orchestrator에 에스컬레이션한다.

---

## 8. 파일 및 코드 변경 원칙

모든 변경은 아래 원칙을 따른다.

### 8.1 최소 변경 원칙

목적 달성에 필요한 최소 범위만 수정한다.

### 8.2 인접 변경 원칙

한 작업에서 변경하는 파일은 가능한 한 관련 파일로 제한한다.

### 8.3 구조 존중 원칙

기존 디렉터리 구조, 빌드 구조, 코드 스타일을 가능한 한 유지한다.

### 8.4 무단 확장 금지 원칙

요청 범위를 넘어 새 기능, 새 정책, 새 설계 방향을 임의로 도입하지 않는다.

### 8.5 변경 근거 명시 원칙

주요 변경은 왜 필요한지 설명 가능해야 한다.

---

## 9. 테스트 및 검증 원칙

코드 또는 설정이 변경된 경우, 가능한 한 검증 절차를 포함해야 한다.

### 9.1 기본 원칙

- 구현 변경에는 검증 근거가 따라야 한다.
- 테스트 가능성이 있는 변경은 테스트 없이 종료하지 않는다.
- 테스트를 수행하지 못한 경우 그 이유를 명시한다.

### 9.2 검증 형태

검증은 상황에 따라 다음 중 하나 이상을 포함할 수 있다.

- 빌드 성공 여부
- 단위 테스트
- 통합 테스트
- 실행 스크립트 검증
- 로그 확인
- 수동 재현 절차 정리

### 9.3 테스트 실패 처리

테스트가 실패한 경우 다음을 남긴다.

- 실패 항목
- 실패 시점
- 실패 원인 추정
- 재현 가능 여부
- 필요한 후속 조치

---

## 10. 기록 원칙

모든 작업은 추적 가능한 기록을 남겨야 한다.

기록에는 가능한 한 아래 항목이 포함되어야 한다.

- 작업 ID
- 작업 목적
- 수행한 내용
- 변경한 파일 또는 대상
- 확인한 근거
- 테스트 또는 검증 결과
- 미해결 이슈
- 현재 상태

작업 결과는 다음 단계 Agent나 Orchestrator가 별도 재해석 없이 이어받을 수 있어야 한다.

---

## 11. 상태 및 실패 기록 원칙

상태 값 자체는 `ORCHESTRATOR.md`를 따르며, 본 문서에서는 상태 기록의 최소 요건만 정의한다.

### 11.1 상태 기록 최소 요건

상태 변경 시 다음을 함께 남긴다.

- 현재 상태
- 상태 변경 이유
- 다음 단계 또는 필요한 조치

### 11.2 실패 기록 최소 요건

실패 또는 차단 상태에서는 다음을 반드시 남긴다.

- 무엇이 실패했는지
- 어디서 실패했는지
- 현재까지 확인한 원인
- 재시도 가능 여부
- 추가 정보 필요 여부

---

## 12. 재작업 원칙

재작업은 단순 반복이 아니라, 실패 원인을 반영한 수정 작업이어야 한다.

재작업 시에는 다음을 명확히 한다.

- 이전 시도에서 무엇이 실패했는지
- 이번 재작업의 목표가 무엇인지
- 어떤 조건이 바뀌었는지
- 어떤 결과가 나오면 다음 단계로 넘길 수 있는지

동일 원인으로 반복 실패하는 경우 무한 반복하지 않고 Orchestrator에 차단 상태를 보고한다.

---

## 13. 에스컬레이션 원칙

아래 상황에서는 Agent가 독자적으로 결정을 내리지 않고 Orchestrator에 에스컬레이션한다.

- 요구사항 충돌
- 범위 불명확
- 설계 문서와 코드가 상충함
- 저장소 구조 변경 필요
- 반복 실패
- 운영 영향 가능성
- 보안 또는 권한 관련 우려
- 역할 범위 초과

에스컬레이션 시에는 아래를 포함한다.

- 현재 작업 요약
- 문제 상황
- 왜 독자 결정이 어려운지
- 가능한 선택지 또는 필요한 추가 정보
- 현재 상태

---

## 14. Repository 작업 기준

Agent는 repository를 단순 참고 자료가 아니라 작업 기준의 단일 출처로 취급한다.

### 14.1 우선 참조 대상

- 현재 사용자 요청
- `PROJECT_STATE.md`
- 운영 문서
- 관련 설계 문서
- 실제 코드와 테스트 코드

### 14.2 문서와 코드 불일치

문서와 코드가 불일치할 경우 다음 원칙을 따른다.

- 현재 사용자 요청을 먼저 확인한다.
- `PROJECT_STATE.md`의 현재 전제를 확인한다.
- 코드가 실제 현재 상태일 가능성을 우선 검토한다.
- 불일치가 중요한 경우 Orchestrator에 보고한다.

---

## 15. Linux daemon 작업 특화 원칙

본 repository의 주요 산출물은 Linux 서버에서 동작하는 Java daemon이므로, 아래 사항을 항상 고려한다.

- 장기 실행 안정성
- 시작 및 종료 흐름
- 예외 처리
- 로그 출력 및 추적성
- 운영 환경 영향
- 재시작 가능성
- 테스트 가능한 구조 유지

기능 구현만으로 충분하다고 보지 않으며, 운영형 소프트웨어로서의 적합성을 함께 고려한다.

---

## 16. 결과 인계 기준

Agent가 작업을 마친 뒤 다음 단계로 넘길 때 결과에는 최소한 아래 내용이 포함되어야 한다.

- 작업 요약
- 수행 내용
- 변경 범위
- 검증 결과
- 미해결 이슈
- 다음 단계 권장 사항

결과 인계 품질이 낮으면 Orchestrator는 보완 후 다시 제출하도록 요구할 수 있다.

---

## 17. 문서 유지 원칙

다음 경우 본 문서를 갱신한다.

- 실제 작업 절차가 변경된 경우
- 허용 또는 제한 작업 범위가 바뀐 경우
- 기록 방식이 변경된 경우
- 테스트 및 실패 처리 방식이 변경된 경우
- Linux daemon 운영 고려사항이 변경된 경우

본 문서는 실행 규칙 문서이므로, 실제 작업 방식이 바뀌면 즉시 함께 수정해야 한다.
