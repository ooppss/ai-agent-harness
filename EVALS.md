# EVALS.md

## 1. 문서 목적

본 문서는 AI Agent 개발 환경의 성능을 평가하기 위한 기준, 시나리오, 측정 항목, 결과 기록 방식을 정의한다.

본 문서의 목적은 다음과 같다.

- 평가 대상을 명확히 한다.
- 성공과 실패의 판단 기준을 통일한다.
- 작업 결과를 정성적 설명이 아니라 측정 가능한 형태로 비교할 수 있게 한다.
- 동일하거나 유사한 작업을 반복 평가할 때 일관된 기준을 유지한다.
- Agent 개발 환경의 개선 여부를 추적할 수 있도록 한다.

본 문서는 기능 요구사항 문서가 아니며, Agent 운영 성과를 평가하기 위한 기준 문서로 사용한다.

---

## 2. 평가 대상

본 문서의 평가 대상은 다음과 같다.

- Orchestrator의 작업 분해 및 배정 품질
- Agent의 구현 품질
- Agent의 테스트 수행 품질
- Agent의 리뷰 및 보고 품질
- 전체 개발 루프의 재현 가능성 및 효율성
- Linux daemon 개발 작업에 대한 적합성

평가는 단일 Agent의 능력만을 보지 않고, Orchestrator와 Agent를 포함한 전체 개발 체계를 대상으로 수행한다.

---

## 3. 평가 범위

현재 평가 범위는 다음과 같다.

- 사용자의 기능 요청을 Orchestrator가 해석할 수 있는가
- 적절한 Agent가 적절한 작업을 수행하는가
- Java daemon 코드가 요구사항에 맞게 구현되는가
- 테스트와 검증이 실제로 수행되는가
- 결과가 사용자에게 이해 가능한 형태로 보고되는가
- 실패 시 원인과 상태가 추적 가능한가

현재 단계에서는 코드 생성 자체보다 **개발 작업 전체의 운영 성능**을 우선 평가한다.

---

## 4. 평가 원칙

평가는 아래 원칙을 따른다.

- 평가 기준은 사전에 정의한다.
- 결과는 가능한 한 측정 가능한 항목으로 기록한다.
- 성공 사례뿐 아니라 실패 사례도 동일한 형식으로 기록한다.
- 단일 작업의 우연한 성공보다 반복 가능한 수행 능력을 더 중요하게 본다.
- 구현, 테스트, 리뷰, 보고를 분리하여 평가한다.
- Linux daemon이라는 제품 특성을 반영한다.

---

## 5. 평가 관점

평가는 아래 관점으로 나누어 수행한다.

### 5.1 요구사항 해석 품질

다음을 평가한다.

- 요청 목적을 정확히 파악했는가
- 범위를 불필요하게 넓히거나 줄이지 않았는가
- 관련 코드 및 문서 범위를 올바르게 식별했는가
- 작업 단위를 적절히 분해했는가

### 5.2 구현 품질

다음을 평가한다.

- Java daemon 코드가 요구사항에 맞게 작성되었는가
- 변경 범위가 과도하지 않은가
- 기존 구조와 일관성이 유지되었는가
- 운영형 소프트웨어로서 기본 요소를 고려했는가

### 5.3 테스트 품질

다음을 평가한다.

- 테스트가 실제로 수행되었는가
- 테스트 범위가 변경 사항과 연결되는가
- 실패 시 재현 정보가 남는가
- 테스트 결과가 다음 단계에 활용 가능한가

### 5.4 리뷰 및 보고 품질

다음을 평가한다.

- 리뷰가 요구사항 충족 여부를 명확히 판단했는가
- 리스크가 식별되고 정리되었는가
- 보고 내용이 간결하면서도 충분한가
- 사용자가 현재 상태를 바로 이해할 수 있는가

### 5.5 전체 작업 흐름 품질

다음을 평가한다.

- Orchestrator가 적절한 Agent를 배정했는가
- 상태 전이가 자연스럽고 추적 가능한가
- 재작업 루프가 무의미하게 반복되지 않았는가
- 실패 시 차단 원인을 명확히 남겼는가

---

## 6. 핵심 평가 지표

현재 기준 핵심 평가 지표는 다음과 같다.

### 6.1 기능 구현 정확도

정의:  
사용자 요청 중 실제로 구현된 항목의 비율

기록 방식 예시:
- 요청 항목 수
- 구현 완료 항목 수
- 누락 항목 수

### 6.2 테스트 수행률

정의:  
구현 변경 중 검증 절차가 수행된 항목의 비율

기록 방식 예시:
- 변경 항목 수
- 테스트 수행 항목 수
- 검증 미수행 항목 수

### 6.3 최초 성공률

정의:  
재작업 없이 한 번에 구현, 테스트, 리뷰를 통과한 작업 비율

기록 방식 예시:
- 전체 작업 수
- 최초 시도 성공 작업 수

### 6.4 재작업 횟수

정의:  
테스트 실패, 리뷰 지적, 범위 재정리 등으로 되돌아간 횟수

기록 방식 예시:
- 작업별 재작업 횟수
- 평균 재작업 횟수

### 6.5 차단 발생률

정의:  
작업이 `blocked` 상태로 전환된 비율

기록 방식 예시:
- 전체 작업 수
- 차단 작업 수
- 차단 원인 분류

### 6.6 보고 완성도

정의:  
최종 보고에 필수 항목이 얼마나 포함되었는지에 대한 수준

필수 항목:
- 요청 요약
- 수행 내용
- 변경 범위
- 테스트 결과
- 리뷰 결과
- 남은 리스크
- 현재 상태

### 6.7 운영 적합성

정의:  
결과물이 Linux daemon으로서 기본 운영 요소를 고려했는지에 대한 평가

확인 항목 예시:
- 시작/종료 흐름 고려 여부
- 로그 고려 여부
- 예외 처리 고려 여부
- 장기 실행 관점 고려 여부

---

## 7. 평가 등급 기준

평가 결과는 다음과 같이 구분한다.

### 7.1 Pass

- 핵심 요구사항이 구현되었다.
- 테스트 또는 검증 결과가 존재한다.
- 리뷰 결과가 정리되었다.
- 남은 리스크가 명시되었다.
- 사용자 보고가 가능한 수준이다.

### 7.2 Partial Pass

- 핵심 요구사항의 일부만 구현되었다.
- 테스트 또는 리뷰가 부분적으로만 수행되었다.
- 결과는 존재하지만 미완료 항목이 명확하다.

### 7.3 Fail

- 핵심 요구사항 구현에 실패했다.
- 테스트 또는 검증이 사실상 수행되지 않았다.
- 결과 기록이 불충분하다.
- 현재 상태와 실패 원인을 명확히 설명할 수 없다.

### 7.4 Blocked

- 구현 자체보다 외부 제약, 요구사항 충돌, 환경 문제 등으로 진행이 중단되었다.
- 차단 원인과 필요한 후속 조치는 기록되어야 한다.

---

## 8. 평가 시나리오 기준

평가는 실제 작업에 가까운 시나리오 단위로 수행한다.

현재 기준 시나리오 유형은 다음과 같다.

- 신규 기능 추가
- 기존 기능 수정
- 버그 수정
- 테스트 보강
- 운영형 개선
- 문서와 코드의 불일치 정리

각 시나리오는 다음 요소를 포함해야 한다.

- 시나리오 ID
- 요청 요약
- 기대 결과
- 관련 컴포넌트
- 난이도
- 평가 항목
- 결과 기록

---

## 9. 결과 기록 형식

각 평가 결과는 아래 형식으로 기록한다.

### 9.1 기본 정보

- 평가 ID
- 수행 일시
- 요청 요약
- 관련 시나리오 유형
- 관련 작업 ID

### 9.2 수행 결과

- 계획 결과
- 구현 결과
- 테스트 결과
- 리뷰 결과
- 최종 보고 결과

### 9.3 지표 기록

- 기능 구현 정확도
- 테스트 수행률
- 재작업 횟수
- 차단 여부
- 보고 완성도
- 운영 적합성

### 9.4 최종 판정

- Pass / Partial Pass / Fail / Blocked
- 주요 사유
- 개선 필요 사항

---

## 10. 결과 해석 원칙

평가 결과는 단순 성공 여부만으로 해석하지 않는다.

다음 원칙을 따른다.

- 구현 성공이라도 테스트와 리뷰가 부족하면 완전한 성공으로 보지 않는다.
- 한 번의 실패가 전체 환경의 실패를 의미하지는 않는다.
- 반복적으로 발생하는 실패 유형은 구조적 문제로 본다.
- 재작업이 있더라도 최종적으로 안정적인 결과를 낼 수 있으면 개선 가능성이 있는 것으로 본다.
- Linux daemon 특성에 대한 고려가 부족하면 일반 코드 생성 성공과 구분하여 해석한다.

---

## 11. 현재 평가 우선순위

현재 평가 우선순위는 다음과 같다.

1. Orchestrator의 작업 분해 및 배정 적절성
2. 구현 결과의 요구사항 충족 여부
3. 테스트 수행 및 실패 추적 가능성
4. 리뷰 및 보고 품질
5. 재작업과 차단의 빈도
6. 운영 적합성

초기 단계에서는 토큰 사용량이나 실행 비용보다 **개발 루프의 안정성과 재현 가능성**을 우선 평가한다.

---

## 12. 평가 결과 활용 원칙

평가 결과는 다음 목적으로 사용한다.

- 운영 문서 개선
- Agent 역할 조정
- Orchestrator 배정 기준 조정
- Harness 실행 규칙 조정
- repository 구조 개선
- 테스트 전략 보완

평가 결과는 단순 기록으로 끝내지 않고, 운영 구조 개선의 근거로 사용한다.

---

## 13. 문서 유지 원칙

다음 경우 본 문서를 갱신한다.

- 평가 지표가 변경된 경우
- 평가 등급 기준이 변경된 경우
- 시나리오 유형이 변경된 경우
- 결과 기록 형식이 변경된 경우
- 프로젝트 목표가 변경된 경우

본 문서는 성능 평가 기준 문서이므로, 평가 방식이 바뀌면 즉시 수정해야 한다.
